# Intel/Altera FPGA SoC Boot Flow — Deep Dive

This article covers the complete boot flow for Intel/Altera **FPGA SoC** and **FPGA/CPLD** families. SoC families with a hard processor: **Cyclone V SoC**, **Arria 10 SoC**, **Agilex 5 SoC** (Cortex-A76/A55), and **Agilex 7 SoC** (Cortex-A53). Non-SoC FPGA with soft processor support: **MAX 10** (CPLD/FPGA — uses soft Nios II or external MCU, not a hard ARM processor). It details the Boot ROM behavior, preloader generation, U-Boot integration, FPGA configuration interfaces, secure boot, and per-board specifics for popular development kits.

---

## Family Overview

### True SoC FPGA Families (Hard Processor System)

These devices contain a **hard processor** (ARM cores in silicon, not FPGA fabric) alongside the FPGA:

| Family | CPU | FPGA Fabric | Boot ROM Size | Key Document |
|---|---|---|---|---|
| **Cyclone V SoC** | ARM Cortex-A9 dual-core @ 925 MHz | Cyclone V FPGA (28 nm) | 64 KB | Intel AN-730 |
| **Arria 10 SoC** | ARM Cortex-A9 dual-core @ 1.2 GHz | Arria 10 GX/SX FPGA (20 nm) | 128 KB | Intel AN-728 |
| **Agilex 5 SoC** | Dual ARM Cortex-A76 + Dual Cortex-A55 @ 1.8 GHz | Agilex 5 FPGA (Intel 7) | 256 KB | Agilex Boot Guide |
| **Agilex 7 SoC** | Quad-core ARM Cortex-A53 @ 1.5 GHz | Agilex 7 FPGA (Intel 7) | 256 KB | Agilex Security Guide |

| Family | Processor | Fabric | Notes |
|---|---|---|---|

## Boot ROM — The First Stage

Intel FPGA SoCs use a **hard processor system (HPS)** Boot ROM that is masked into on-chip ROM during manufacturing. It is not field-upgradable.

### Boot ROM Behavior

1. **Power-on reset (POR)**: All HPS clocks are released from reset. The Boot ROM starts executing from its internal ROM at a fixed address (typically `0xFFFF0000`).
2. **BSEL sampling**: The Boot ROM samples the **BSEL[2:0]** pins to determine boot source.
3. **Clock setup**: The Boot ROM configures the main PLL to a safe frequency (typically 25–50 MHz).
4. **Load preloader**: The Boot ROM reads the preloader from the selected boot medium into on-chip RAM (OCRAM — 64 KB for Cyclone V, 128 KB for Arria 10, 256 KB for Agilex).
5. **Authentication (if secure boot enabled)**: The Boot ROM verifies the preloader signature using the root public key hash stored in eFuse.
6. **Jump to preloader**: The Boot ROM transfers control to the preloader at the OCRAM entry point.

### BSEL Pin Decoding Truth Table

| BSEL[2:0] | Boot Source | Cyclone V | Arria 10 | Agilex 5/7 |
|---|---|---|---|---|
| `000` | Reserved | — | — | — |
| `001` | FPGA configuration first | Yes | No | No |
| `010` | NAND flash (8-bit) | Yes | No | No |
| `011` | SD/MMC card | Yes | Yes | Yes |
| `100` | QSPI flash | Yes | Yes | Yes |
| `101` | eMMC | No | Yes | Yes |
| `110` | JTAG / UART download | Yes | Yes | Yes |
| `111` | External (FPGA-controlled) | Yes | No | No |

> **Cyclone V BSEL trap**: On many dev boards (DE10-Nano, SoCKit), BSEL pins are wired to DIP switches or pull-down resistors. If all switches are OFF, BSEL = `000` (Reserved) — the Boot ROM hangs silently. Always verify BSEL in schematics.

### Reading BSEL from Software

```bash
# Linux sysfs (Cyclone V)
cat /sys/kernel/debug/socfpga/bootinfo
# Output: BOOT_FROM_SDMMC (or NAND, QSPI, etc.)

# Direct register read (Cyclone V sysmgr.bootinfo at 0xFFD08010)
devmem 0xFFD08010
# Bits [2:0] = BSEL value
```

## Preloader — The First User Code

Intel provides two paths for the preloader:

1. **Intel Preloader** (legacy, generated by BSP Editor): A binary blob generated from the QSys/Platform Designer handoff. Proprietary, limited customization.
2. **U-Boot SPL** (modern, open-source): The standard U-Boot first-stage loader, customized for the `socfpga` architecture. Full source control.

> **Recommendation**: Use U-Boot SPL for all new designs. It is actively maintained, supports modern Linux kernels, and integrates with distro boot.

### Preloader Image Format (`mkpimage`)

Intel Boot ROM expects the preloader in a special format called **mkpimage** (masked ROM program image). The `mkpimage` tool (part of Intel SoC EDS) wraps the raw binary with:

- Header with validation words
- ECC/checksum for Boot ROM verification
- Multiple copies for failover (optional)

```bash
# Generate preloader image from U-Boot SPL binary
mkpimage -o preloader-mkpimage.bin \
         -h 1024 \
         -e 0xFFFF0000 \
         u-boot-spl.bin

# For Cyclone V with multiple copies (SD boot failover)
mkpimage -o preloader-mkpimage.bin \
         -h 1024 \
         -e 0xFFFF0000 \
         -c 2 \
         u-boot-spl.bin u-boot-spl.bin
```

| Parameter | Meaning |
|---|---|
| `-h 1024` | Header size: 1024 bytes |
| `-e 0xFFFF0000` | Entry point address (Boot ROM jump target) |
| `-c 2` | Number of copies (for failover) |

### Cyclone V Preloader on SD Card

The Cyclone V Boot ROM reads the preloader from SD card in one of two ways:

1. **Raw partition (A2 partition)**: A special partition type `0xA2` at the start of the SD card. The Boot ROM reads blocks directly from this partition.
2. **FAT partition**: The preloader is stored as a file in a FAT32 partition. Less common.

```bash
# Create A2 partition for Cyclone V preloader
sudo parted /dev/sdX --script -- mkpart primary 0xA2 0 2MB

# Write preloader to A2 partition
sudo dd if=preloader-mkpimage.bin of=/dev/sdX1 bs=1M
```

### U-Boot SPL for SoCFPGA

```c
// Key SPL functions in arch/arm/mach-socfpga/spl.c

void board_init_f(ulong dummy) {
    // 1. Enable clocks (main, peripheral, SDRAM PLLs)
    socfpga_per_reset_assert();
    socfpga_bridges_reset();
    
    // 2. Configure pin muxing from handoff data
    sysmgr_pinmux_init();
    
    // 3. Initialize SDRAM (DDR calibration)
    //    This reads the handoff DDR parameters from QSys
    sdram_calibration();
    
    // 4. Configure FPGA (if CONFIG_SPL_FPGA)
    #ifdef CONFIG_SPL_FPGA
    fpga_load(0, bitstream_addr, bitstream_size);
    #endif
    
    // 5. Load U-Boot proper from SD/FAT or QSPI
    spl_load_image_fat_os(boot_dev);
}
```

## U-Boot — The Secondary Bootloader

U-Boot proper for Intel SoC is built with the `socfpga` board support package. It is a fully-featured bootloader with filesystem, network, and FPGA loading support.

### Building U-Boot for Cyclone V

```bash
# Clone U-Boot
git clone https://github.com/u-boot/u-boot.git
cd u-boot

# Configure for DE10-Nano (Cyclone V)
make ARCH=arm socfpga_de10_nano_defconfig

# Build
make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- -j$(nproc)

# Outputs:
#   u-boot-spl.bin  →  SPL (first stage)
#   u-boot.img      →  U-Boot proper (second stage)
```

### U-Boot FPGA Commands (Intel SoC)

```bash
# Load bitstream from FAT partition on SD
fatload mmc 0:1 0x2000000 core.rbf
fpga load 0 0x2000000 ${filesize}

# Load peripheral RBF (Arria 10/Agilex split config)
fatload mmc 0:1 0x2000000 peripheral.rbf
fpga load 0 0x2000000 ${filesize}

# Check FPGA status
fpga info 0
# Output: FPGA[0] state: Powered Up / Configured

# For Arria 10/Agilex — use bridge reset after config
bridge enable
```

### Boot Script (`boot.scr`)

```bash
# boot.cmd — source file for mkimage
# Generate with: mkimage -A arm -T script -C none -n "boot" -d boot.cmd boot.scr

# 1. Load bitstream
fatload mmc 0:1 0x2000000 core.rbf
fpga load 0 0x2000000 ${filesize}

# 2. Load device tree
fatload mmc 0:1 ${fdtaddr} socfpga_cyclone5_socdk.dtb

# 3. Load kernel
fatload mmc 0:1 ${kernel_addr_r} zImage

# 4. Set bootargs
setenv bootargs console=ttyS0,115200 root=/dev/mmcblk0p2 rw rootwait

# 5. Boot Linux
bootz ${kernel_addr_r} - ${fdtaddr}
```

### Distro Boot Support

Modern U-Boot for SoCFPGA supports distro boot:

```bash
# U-Boot scans these files in order on each boot medium:
# 1. /extlinux/extlinux.conf
# 2. /boot/extlinux/extlinux.conf
# 3. /boot.scr.uimg
# 4. /boot.scr

# Example extlinux.conf for DE10-Nano
LABEL primary
    MENU LABEL Primary Kernel
    LINUX ../zImage
    FDT ../socfpga_cyclone5_socdk.dtb
    APPEND console=ttyS0,115200 root=/dev/mmcblk0p2 rw rootwait
```

## FPGA Configuration Interfaces

Intel SoCs support multiple FPGA configuration interfaces depending on the family.

| Family | Interface | Width | Speed | Use Case |
|---|---|---|---|---|
| **Cyclone V** | FPP (Fast Passive Parallel) | 16-bit | ~25 MB/s | HPS drives config pins directly |
| **Arria 10** | HPS-to-FPGA bridge | 32-bit | ~100 MB/s | HPS writes to config registers via bridge |
| **Agilex** | AVST (Avalon Streaming) + HPS bridge | 32-bit | ~200 MB/s | Streaming config + random access |
| **MAX 10** | Internal flash | N/A | N/A | FPGA configured from internal flash at power-on |

### FPP x16 (Cyclone V)

The HPS drives the FPGA configuration pins (`DCLK`, `DATA[15:0]`, `nCONFIG`, `nSTATUS`, `CONF_DONE`) directly. This is a passive parallel interface:

1. HPS asserts `nCONFIG` low
2. HPS waits for `nSTATUS` high
3. HPS toggles `DCLK` and drives `DATA[15:0]` with bitstream bytes
4. HPS waits for `CONF_DONE` high
5. HPS sends two additional `DCLK` cycles

> **DE10-Nano / MiSTer**: The FPP interface is used by both U-Boot SPL and the Linux FPGA Manager driver. The `.rbf` file is the raw bitstream without any header — just the configuration frames.

### Arria 10 / Agilex — HPS-to-FPGA Bridge

Newer families use a memory-mapped bridge for FPGA configuration:

```c
// Arria 10 FPGA config via HPS bridge
// Write bitstream to config data register
writel(bitstream_word, SOCFPGA_FPGMGR_DATA);

// Poll status
while (!(readl(SOCFPGA_FPGMGR_STAT) & FPGMGR_STAT_CONF_DONE))
    ;
```

## SD Card Layout for Intel Dev Boards

### DE10-Nano / MiSTer Standard Layout

```
/dev/mmcblk0  (8+ GB SD card)
├─ /dev/mmcblk0p1  [A2 partition, 2 MB, raw]
│   └─ preloader-mkpimage.bin (raw, at offset 0)
├─ /dev/mmcblk0p2  [FAT32, 256 MB, label: BOOT]
│   ├─ u-boot.img
│   ├─ zImage
│   ├─ socfpga_cyclone5_socdk.dtb
│   ├─ core.rbf
│   └─ boot.scr
└─ /dev/mmcblk0p3  [ext4, remainder, label: rootfs]
    └─ Full Linux root filesystem
```

### SoCKit / Arrow SoC Development Kit

```
/dev/mmcblk0
├─ /dev/mmcblk0p1  [FAT32, 256 MB]
│   ├─ preloader-mkpimage.bin  (or in A2 partition)
│   ├─ u-boot.img
│   ├─ zImage
│   ├─ socfpga_cyclone5_socdk.dtb
│   └─ boot.scr
└─ /dev/mmcblk0p2  [ext4, remainder]
    └─ rootfs
```

> **MiSTer convention**: The MiSTer framework dynamically loads `.rbf` files from `/media/fat/_Core/` via the Linux FPGA Manager. The kernel boots first, then userspace loads the specific core bitstream. This is the "userspace-triggered" FPGA config pattern.

## Secure Boot on Intel SoCs

### Cyclone V / Arria 10 Secure Boot

1. **AES-256 bitstream encryption**: The FPGA bitstream is encrypted with a user key. The key is stored in battery-backed RAM (BBRAM) or eFuse.
2. **HMAC-SHA-256 authentication**: The preloader and U-Boot are authenticated using HMAC-SHA-256. The key is stored in eFuse.
3. **eFuse programming**: One-time programmable. Stores:
   - Root public key hash (for signature verification)
   - AES key (for bitstream decryption)
   - Security policy bits (disable JTAG, force secure boot)

```bash
# Program eFuses (IRREVERSIBLE — test first!)
quartus_pgm -c 1 -m JTAG -o "p;secure_key.sof"

# Generate encrypted bitstream
quartus_cpf -c -o bitstream_encryption=on design.sof design_encrypted.rbf
```

### Agilex Secure Boot (Next-Generation)

Agilex uses a modern security stack:

- **ECDSA P-384** for bootloader authentication
- **AES-256-GCM** for bitstream encryption
- **PUF (Physically Unclonable Function)** for key derivation
- **Anti-rollback** counters in eFuse
- **Full debug disable** capability

```bash
# Agilex secure boot image generation
quartus_pgm -c 1 -m JTAG -o "p;agilex_secure_boot.sof"

# Signed bitstream
quartus_sign design.rbf -k signing_key.pem -o signed_design.rbf
```

### Security Policy Bits

| eFuse Bit | Effect |
|---|---|
| `FORCE_SECURE_BOOT` | Boot ROM rejects unauthenticated preloaders |
| `DISABLE_JTAG` | JTAG boundary scan disabled. Recovery only via Boot ROM UART |
| `DISABLE_EXTERNAL_CONFIG` | FPGA can only be configured by HPS (not external master) |
| `AES_KEY_SOURCE` | Select BBRAM vs eFuse for AES key storage |

> **Warning**: Once `DISABLE_JTAG` is programmed, you cannot use JTAG for debugging or recovery. Ensure your UART boot path is tested before enabling this.

## Common Intel-Specific Boot Problems

### "Boot ROM hangs with no UART output"

| Check | Action |
|---|---|
| BSEL pins | Verify with DMM. Default should be `011` (SD) on most boards |
| Preloader in A2 partition | `hexdump -C /dev/sdX1 | head` — should start with Intel mkpimage header (`0x31305341` = "AS01") |
| SD card speed | Try a slower SD card (Class 4). Boot ROM may fail with UHS-I cards |
| Power sequencing | Scope 3.3V, 1.8V, 1.1V rails. CPU core must be stable before POR release |

### "U-Boot loads but kernel hangs at 'Starting kernel...'"

| Cause | Fix |
|---|---|
| Wrong device tree | Ensure `.dtb` matches the actual FPGA configuration (HPS IO muxing) |
| DDR size mismatch | `bootargs` must match actual DDR size. Check `mem=1024M` parameter |
| Missing `rootwait` | SD card detection takes time. Always use `rootwait` in `bootargs` |
| FPGA not configured | If kernel drivers expect FPGA peripherals, load `.rbf` in U-Boot before `bootz` |

### "FPGA Manager fails with 'write init error'"

| Cause | Fix |
|---|---|
| Bridge not released | Run `echo 0 > /sys/class/fpga_manager/fpga0/flags` before loading |
| Wrong `.rbf` format | Cyclone V needs **peripheral + core** RBF for full config. Use `quartus_cpf` to generate correctly |
| FPGA already configured | Check `cat /sys/class/fpga_manager/fpga0/state` — if `operating`, reset FPGA first |
